# Reference CLI 工具 - 需求规格说明书

## 1. 项目概述

### 1.1 项目名称
`reference`

### 1.2 项目定位
面向 AI 辅助编程时代的本地代码仓库引用管理器。通过统一的全局缓存和项目级 Junction 链接机制，让开发者及 AI 助手能够以零网络延迟、零上下文污染的方式查阅任意远程或本地 Git 仓库的代码实现。

### 1.3 核心价值
- **开发者体验**：将手工 `mkdir`、`git clone`、软链接、`.gitignore` 维护等操作压缩为一条命令或一句对话。
- **AI 协同效率**：通过自动生成的 Skill 和子代理架构，让 AI 理解项目可用的参考资料，并能主动添加缺失的仓库。
- **知识积累与复用**：全局 wiki 一次生成、跨项目复用。问题探索结果沉淀为主题文件，后续遇到同类问题直接复用。
- **资源安全边界**：全局缓存供多项目共享，严格保护用户本地目录（本地引用只管理链接，绝不删除源文件）。

## 2. 使用场景与用户故事

### 2.1 远程仓库引用
- **场景**：开发者需要参考 `gin-gonic/gin` 的路由实现。
- **操作**：执行 `reference add https://github.com/gin-gonic/gin`。
- **结果**：工具自动浅克隆到全局缓存，创建 Junction 链接，更新 `.gitignore`，自动注入 AI Skill 文件。

### 2.2 本地仓库引用
- **场景**：开发者正在本地开发一个共享库 `~/projects/my-lib`，希望在另一个项目中参考其代码。
- **操作**：执行 `reference add --local ~/projects/my-lib --name my-utils`。
- **结果**：校验为 Git 仓库后创建 Junction 链接，AI 即可正常识别和搜索。

### 2.3 AI 对话式自动管理
- **场景**：在 Claude Code 中对话："我想参考一下 `spf13/cobra` 的命令行设计"。
- **过程**：AI 检测到本地无此仓库，主动调用 `reference add` 下载，然后分析并返回结果。

### 2.4 知识积累与复用
- **场景**：首次探索"forks 仓库的代理实现"，生成 `代理实现.md`。后续在其他项目中再次需要参考，直接复用已有分析。
- **过程**：AI 先检查知识目录，已有主题文件直接返回，没有则探索后写入供后续复用。

### 2.5 清理与维护
- `reference remove <id>`：仅删除链接。
- `reference remove <id> --purge`：二次确认后删除全局缓存。
- 对本地引用 `--purge` 被忽略并提示。

## 3. 系统架构设计

### 3.1 全局缓存与 Wiki 架构
```
~/.cicbyte/reference/
├── config/config.yaml      # 用户配置
├── db/app.db               # SQLite 数据库（纯 Go，无 CGO）
├── repos/                  # 全局缓存（可配置）
│   └── github.com/cicbyte/forks/
├── wiki/                   # 全局知识库（一次生成，跨项目复用）
│   └── github.com-cicbyte-forks/
│       ├── reference.md    # 仓库结构总览
│       └── 代理实现.md      # 主题知识文件
└── logs/                   # 日志
```

### 3.2 项目级引用目录
```
<project>/
├── .reference/
│   └── github.com-cicbyte-forks -> ~/.cicbyte/reference/repos/github.com/cicbyte/forks
└── .claude/
    └── skills/reference/
        ├── SKILL.md                              # 仓库地图 + 查询策略
        ├── reference-explorer.md                 # 子代理：知识探索
        ├── reference-analyzer.md                 # 子代理：深度分析
        └── github.com-cicbyte-forks -> ~/.cicbyte/reference/wiki/...  # Junction 到全局 wiki
```

### 3.3 AI Agent 集成组件
| 组件 | 路径 | 作用 |
| :--- | :--- | :--- |
| **Skill 文件** | `.claude/skills/reference/SKILL.md` | 主模型指令，含仓库地图、知识优先查询策略 |
| **Explorer 子代理** | `.claude/agents/reference-explorer.md` | 知识探索：检查已有知识 → 探索代码 → 写入主题文件 |
| **Analyzer 子代理** | `.claude/agents/reference-analyzer.md` | 深度分析：生成完整 reference.md（全局一次） |

### 3.4 知识体系设计
- **reference.md** — 仓库结构总览，深度分析一次，全局复用
- **`<主题>.md`** — 特定问题探索结果（如 `代理实现.md`），按需积累，跨项目复用
- 知识文件用**函数名**定位（禁止行号，行号随更新失效），复杂逻辑使用 Mermaid 图

## 4. 命令行接口 (CLI) 详细规范

### 4.1 `reference add <target> [flags]`
| 参数 | 描述 |
| :--- | :--- |
| `<target>` | Git URL（远程）或本地路径（`--local`） |
| `--local` `-l` | 标记为本地 Git 仓库路径 |
| `--name` `-n` | 自定义链接名称 |
| `--branch` `-b` | 指定克隆分支（仅远程） |
| `--depth` `-d` | 浅克隆深度，默认 1（仅远程） |
| `--no-update` | 缓存已存在时跳过 git pull（仅远程） |

**执行流程**：
1. 解析 URL 或校验本地路径
2. 克隆/拉取到全局缓存（远程）或直接使用（本地）
3. 创建 `.reference/` 下的 Junction 链接
4. 更新 `.gitignore`
5. 写入数据库索引
6. 自动调用 `inject` 生成 AI Agent 配置

### 4.2 `reference remove <identifier> [flags]`
| 参数 | 描述 |
| :--- | :--- |
| `<identifier>` | 链接名称或 ID |
| `--purge` | 同时删除全局缓存（远程），需确认；本地引用忽略 |

### 4.3 `reference list`
列出当前项目所有引用，显示类型、链接名、目标、commit、分支。

### 4.4 `reference update [<identifier>]`
更新指定或全部远程引用（git pull），跳过本地引用。

### 4.5 `reference inject`
扫描 `.reference/` 目录，生成 AI Agent 配置：
- 渲染 SKILL.md（仓库地图 + 查询策略）
- 为每个仓库创建 Junction 链接到全局 wiki
- 复制子代理配置文件

## 5. 技术实现要点

### 5.1 技术栈
- 语言：Go 1.24+
- CLI 框架：`github.com/spf13/cobra`
- Git 操作：`github.com/go-git/go-git/v5`
- 数据库：`github.com/glebarez/sqlite`（纯 Go，无 CGO）
- 代理：`golang.org/x/net/proxy`（SOCKS5）、`net/http`（HTTP）

### 5.2 CMD/Logic 分层（cobra-app 规范）
- **cmd/root.go** — 唯一根命令
- **cmd/repo/** — repo 模块 CMD 层（参数绑定、验证、调用 Logic）
- **internal/logic/repo/** — repo 模块 Logic 层（纯业务逻辑，不依赖 cobra）
- 两层通过 `*Config` 结构体交互，Logic 层通过工厂方法创建 Processor，入口为 `Execute(ctx)`

### 5.3 跨平台链接
- Windows：`New-Item -ItemType Junction`（PowerShell，无需管理员权限）
- Unix-like：`os.Symlink`

### 5.4 网络代理
- 配置：`config.yaml` 中 `network.proxy` / `network.git_proxy`
- 优先级：`git_proxy` > `proxy` > 环境变量 `HTTPS_PROXY` > 无代理
- HTTP 代理：`http.Transport{Proxy: http.ProxyURL(parsed)}`
- SOCKS5 代理：`golang.org/x/net/proxy.SOCKS5` → `http.Transport{DialContext}`
- 通过 `client.InstallProtocol` 注册到 go-git，操作完成后恢复默认 transport

### 5.5 应用初始化流程（cmd/root.go init()）
严格按顺序：`InitAppDirs` → `LoadConfig` → `InitLog` → `GetGormDB`（含 AutoMigrate），任何步骤失败 `os.Exit(1)`。
